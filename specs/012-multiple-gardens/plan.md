# Implementation Plan: Multiple Gardens (012)

**Branch**: `012-multiple-gardens` | **Date**: 2026-03-05 | **Spec**: `specs/012-multiple-gardens/spec.md`
**Input**: Feature description — multiple named garden plans per user, three creation methods (new, copy from existing, import CSV)

## Summary

Users can create multiple named garden plans (e.g., one per year or season) and switch between them. Each garden independently holds its beds, dimensions, and map layout. Three creation methods: blank, clone an existing garden, or import bed structure from CSV. All existing pages (Map, Beds, Dashboard) automatically scope to the active garden.

## Technical Context

**Language/Version**: Node.js 22 (backend) / React 19 + JSX (frontend)
**Primary Dependencies**: Express 5, Mongoose 9, TanStack React Query 5, React Router 7, Tailwind CSS 3, Lucide React, csv-parse 6.x (already installed), multer v2 (already installed)
**Storage**: MongoDB 7 — new `gardens` collection; additive `gardenId` field on `GardenBed`; additive `activeGardenId` on `User`
**Testing**: Jest 29 + Supertest + mongodb-memory-server (backend); Playwright 1.x (E2E)
**Target Platform**: Linux server (Docker), browser
**Project Type**: Web application (backend API + React SPA)
**Performance Goals**: No new perf requirements; garden list is small (< 50 gardens per user typically)
**Constraints**: All existing beds must be migrated to a default garden before new code runs. No API response shape regressions without flagging.
**Scale/Scope**: Single-product app; migration runs once per deployment

## Implementation Approach

### Phase A — Backend (data + API)

**A1 — Garden model** (`backend/src/models/Garden.js`)
```js
const gardenSchema = new mongoose.Schema({
  userId:       { type: ObjectId, ref: 'User', required: true, index: true },
  name:         { type: String, required: true, trim: true },
  gardenWidth:  { type: Number, min: 1, default: null },
  gardenHeight: { type: Number, min: 1, default: null },
  gardenImage:  { type: String, default: null },
}, { timestamps: true });
```

**A2 — GardenBed model update**
Add `gardenId: { type: ObjectId, ref: 'Garden', required: true, index: true }`.
Note: `required: true` is safe because migration runs before deploying new code.

**A3 — User model update**
Add `activeGardenId: { type: ObjectId, ref: 'Garden', default: null }`.

**A4 — Migration script** (`backend/src/seed/migrateToMultiGarden.js`)
- Reads all `owner` users where `activeGardenId` is null.
- Creates a `Garden` for each, sets `user.activeGardenId`, bulk-updates matching `GardenBed` docs.

**A5 — gardens.js route** — all endpoints from the API contract:
- `GET /api/gardens` — list own gardens.
- `POST /api/gardens` — create (blank or clone from `sourceGardenId`).
- `POST /api/gardens/import` — multer single-file + csv-parse, create garden + beds.
- `PUT /api/gardens/:id` — update name/dimensions (with placed-bed clip check).
- `POST /api/gardens/:id/image` — multer image upload.
- `DELETE /api/gardens/:id/image` — remove image file.
- `DELETE /api/gardens/:id` — cascade-delete beds; refuse if last garden.
- `PUT /api/auth/me/active-garden` — set activeGardenId on User (in `auth.js`).

**A6 — auth.js changes**
- `buildUserPayload(user)` — async, populates active garden, includes `activeGardenId`, falls back to User fields for unmigrated accounts.
- All callers `await buildUserPayload(user)`.
- Add `PUT /api/auth/me/active-garden` route.

**A7 — beds.js changes**
- `GET /api/beds`: scope filter to `{ userId, gardenId }`. `gardenId` from `req.query.gardenId`. Return `400` if `gardenId` absent and user has gardens.
- `POST /api/beds`: require `gardenId` in body; validate it belongs to `req.gardenOwnerId`.
- `PUT /api/beds/:id` and `DELETE /api/beds/:id`: dimension/overlap checks use `garden.gardenWidth/gardenHeight` from `Garden.findById(bed.gardenId)` instead of `User.gardenWidth/gardenHeight`.

**A8 — access.js changes**
`/api/access/shared`: populate `ownerId.activeGardenId` and include in response.

**A9 — app.js**: mount `require('./routes/gardens')` at `/api/gardens`.

**A10 — Test updates** (`helpers.js`, `beds.test.js`, new `gardens.test.js`)

### Phase B — Frontend

**B1 — GardenContext.jsx**
New state:
- `gardens: Garden[]` — fetched from `/api/gardens` on mount.
- `currentGardenId: string|null` — `activeGardenId` from user object; updated when user switches.
Expose `setCurrentGardenId(id)` — calls `PUT /api/auth/me/active-garden`, updates context + `gh_user` localStorage.
For helpers: `currentGardenId` comes from `activeGarden.activeGardenId`.

**B2 — GardenBeds.jsx**
All query/mutation calls add `gardenId`:
```js
queryFn: () => api.get(`/beds?gardenId=${currentGardenId}`).then(r => r.data)
// POST body includes gardenId
```

**B3 — GardenMap.jsx**
- Pass `gardenId` to bed queries.
- Read garden dimensions from context (`useGarden().currentGarden`) instead of `user.gardenWidth`.

**B4 — GardenDimensionsModal.jsx**
- Call `PUT /api/gardens/:currentGardenId` instead of `PUT /api/auth/me/garden`.
- On success, invalidate `['gardens']` query + update `gh_user` localStorage.

**B5 — AppLayout.jsx**
Add own-garden switcher above the existing shared-garden switcher:
- Dropdown listing user's own gardens; selected = `currentGardenId`.
- "+ New" button → opens `NewGardenModal`.
- When switching own garden: call `setCurrentGardenId`, clear React Query cache.

**B6 — NewGardenModal.jsx** (new component)
Three tabs: New / Copy / Import CSV.
- **New**: name input + optional width/height → `POST /api/gardens`.
- **Copy**: name input + garden select dropdown → `POST /api/gardens` with `sourceGardenId`.
- **Import CSV**: name input + file picker → `POST /api/gardens/import` (multipart).
On success: invalidate `['gardens']`, switch active garden to new one.

**B7 — Profile.jsx**
Remove the garden name/dimensions/image section (now per-garden, managed via garden switcher).

## Risk Notes

1. **Migration must run before deploy** — if skipped, `gardenId` required validation on GardenBed will reject all bed creates.

2. **Existing E2E tests** — `beds.spec.js`, `gardenMap.spec.js`, `dashboard.spec.js` will need minor updates since bed API calls now require `gardenId`. Global Playwright setup creates the fixture user; after migration that user will have a garden auto-created.

3. **`PUT /api/beds/:id` overlap check** — currently reads `owner.gardenWidth` from User. Must be updated to read from the bed's Garden document.
